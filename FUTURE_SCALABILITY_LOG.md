# 技術前進ログ：スケーラビリティとパフォーマンス (Scalability & Performance)

## 現状の設計 (v1.3.x)
- **集計・ソート**: データベース（Firestore）側のインデックス要件を回避し開発スピードを優先するため、サーバーサイド（Cloud Functions）のメモリ上で集計およびソートを行っています。
- **対象**: クライアントダッシュボードのジョブ履歴、ポイント取引履歴など。

## 将来の課題 (ユーザー数 10,000名規模への到達時)
ユーザー数および 1 ユーザーあたりの履歴データが膨大になった場合、以下の問題が発生する可能性があります：
1. **通信量の増大**: 全履歴取得によるサーバー・DB間のトラフィック増加。
2. **サーバーリソース消費**: 大量データのパースおよびメモリ上での計算負荷。

## 推奨される改善プロセス
サービスが成長し、スケーラビリティが優先されるフェーズに移行する際、以下の手順で最適化を行います：

1. **インデックスの作成**:
   - Google Cloud Console および Firebase Console にて、頻繁に使用されるクエリ（`where: { userId }, orderBy: { createdAt: 'desc' }` 等）に対する複合インデックスを作成する。
2. **DBレベル集計への移行**:
   - `Prisma` または Firestore SDK の `aggregate` / `count` 機能を使用し、DB側で計算を完結させる。
3. **パジネーションの強化**:
   - メモリ上での `slice` ではなく、DBレベルでの `take` / `skip` (Cursor-based) に完全移行する。
